# Windsurf Rules — Telegram Bot для мастера по наращиванию ресниц

## Общие правила проекта

### Язык и стиль
- Весь код на Python 3.10+
- Комментарии в коде на русском языке
- Названия переменных и функций на английском языке
- Docstrings на русском языке
- Использовать type hints везде, где возможно

### Архитектура проекта
- Модульная структура: каждый логический блок в отдельном файле
- Обработчики (handlers) разделены по функционалу: запись, админ-панель, общие команды
- База данных — отдельный модуль с асинхронными функциями
- Конфигурация вынесена в config.py
- Клавиатуры (keyboards) в отдельном модуле

### Технологический стек
- **aiogram 3.x** (последняя стабильная версия) — НЕ aiogram 2.x
- **aiosqlite** — асинхронная работа с SQLite
- **APScheduler** (AsyncIOScheduler) — планировщик задач для напоминаний
- **Python 3.10+**

### Правила работы с aiogram
- Использовать Router вместо Dispatcher для регистрации обработчиков
- FSM через StatesGroup и State из aiogram.fsm
- Inline-кнопки через InlineKeyboardBuilder или InlineKeyboardMarkup
- Callback data через CallbackData factory где это уместно
- HTML-разметка для форматирования сообщений (parse_mode=HTML)

### Правила работы с базой данных
- Все операции с БД асинхронные (aiosqlite)
- При первом запуске создавать таблицы автоматически
- Таблицы: users, slots, bookings, work_days
- Использовать параметризованные запросы (защита от SQL-инъекций)
- Каждая функция БД — отдельная async-функция с понятным именем

### Правила FSM
- Состояния для записи: выбор даты → выбор времени → ввод имени → ввод телефона → подтверждение
- Состояния для админки: отдельная группа состояний
- Всегда предусматривать возможность отмены (кнопка "Назад" или "Отмена")

### Правила безопасности
- Токен бота НИКОГДА не хардкодить — только через config.py или переменные окружения
- ID администратора проверять в каждом обработчике админ-панели
- Проверка подписки на канал перед доступом к записи

### Правила форматирования сообщений
- Использовать HTML-теги: <b>, <i>, <code>, <a>
- Эмодзи для визуального оформления: 📅 🕐 ✅ ❌ 📞 👤 💰 📋
- Структурированные сообщения с отступами и разделителями

### Обработка ошибок
- Try-except в критических местах (работа с БД, API Telegram)
- Логирование ошибок через модуль logging
- Graceful degradation — бот не должен падать при ошибке

### Планировщик задач (APScheduler)
- Напоминания за 24 часа до записи
- При отмене записи — удалять запланированное напоминание
- При старте бота — восстанавливать задачи из БД
- Если до записи менее 24 часов — напоминание не создавать
- ID задачи формировать из booking_id для лёгкого управления

### Проверка подписки
- Использовать метод getChatMember
- Статусы "member", "administrator", "creator" считать подписанным
- Middleware или отдельная проверка перед бронированием

### Ограничения бизнес-логики
- Один пользователь = максимум одна активная запись
- Расписание формируется на 1 месяц вперёд
- При бронировании слот становится недоступным
- При отмене — слот снова доступен
- Уведомление админу + в канал при каждой новой записи

## Что НЕ делать
- Не использовать polling в продакшн-режиме без необходимости (но для MVP — допустимо)
- Не хранить состояние в глобальных переменных
- Не использовать синхронные библиотеки для БД
- Не игнорировать обработку ошибок
- Не делать один огромный файл — разбивать по модулям
